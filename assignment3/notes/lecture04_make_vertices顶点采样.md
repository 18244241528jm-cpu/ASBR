## 2. 概念：顶点是什么？

- **构型（configuration）**：规划组里每个 **可控关节** 一个角度（或位移），组成向量 **q**。  
- **顶点**：一个 **无碰撞** 的 **q**，在 PRM 里作为图上的一个 **节点**。  
- **构型空间的一个盒子**：作业 PDF 写明每个关节 **∈ [−π, π]**。实现时以 **`assignment3_context` 头文件注释** 为准；若注释与 URDF 实际限位不一致，**作业文本/教师说明优先**。

**注意**：真实 UR5 的某些关节物理范围可能小于 ±π；若课程骨架 **明确要求 ±π**，按作业要求采样，MoveIt 可能在 `RobotState` 里 **enforce bounds** —— 读骨架里 `state_collides` 前是否已夹紧关节，避免「采样在界外被 silently 改掉」导致困惑。

## 3. 采样策略：拒绝采样（rejection sampling）

**与 Week05 PPT 的对应**：幻灯里用 **Sample → Discard（碰撞）→ Keep（自由）→ Repeat** 演示路图顶点从何而来；你在 **`make_vertices`** 里做的正是 **在构型空间盒内均匀采样 + 碰撞则拒绝**，与「用样本点近似 \(Q_{\mathrm{free}}\)」一致。PPT 还提醒：**旋转关节要「公平」地采样** —— 本作业用 **各关节 \([-\pi,\pi]\) 上均匀随机** 即是一种常见做法（维数 = 规划组关节数）。

最简单且足够交作业的做法：

1. 在 **每个** 规划关节上 **均匀随机** 取 **[-π, π]**（或作业指定的分布）。  
2. 把这一组值写进 **`RobotState`**（或骨架提供的状态对象）。  
3. 调用 **`state_collides(...)`**；若为 **真**（碰撞），**丢弃**，回到步骤 1。  
4. 若为 **假**，将该状态 **加入 V**。  
5. 重复直到 **|V| = n**（n 为目标顶点数，由你选成员变量或常量；太小图不连通，太大变慢）。

**伪代码**（语言无关）：

```text
V ← 空列表
attempts ← 0
while |V| < n and attempts < max_attempts:
    attempts ← attempts + 1
    for 每个规划关节 j:
        q[j] ← Uniform(-π, π)
    把 q 写入 RobotState（或等价）
    if not state_collides(该状态):
        V.append(该状态的副本)
若 |V| == 0: 返回失败（或由 solve 统一处理）
```

## 4. 和 MoveIt 打交道时要注意什么？（概念级）

骨架不同，细节不同，但通常涉及：

1. **关节名字与顺序**：不能自己臆造顺序；应来自 **`JointModelGroup`**（或骨架已给的迭代方式）。  
2. **写关节值**：常用思路是对 **组内每个关节** 调用设置变量位置的方法（具体 API 名查 MoveIt 文档 **RobotState**）。  
3. **副本**：`RobotState` 往往要用 **`copy`** 或构造新状态再写入，避免多个顶点 **共享同一块被后续修改的内存**。  
4. **碰撞检测用的场景**：必须与规划时一致；骨架里 `state_collides` 一般已绑好 **Planning Scene**，你只需传入 **正确填充** 的状态。

**小白检查**：采完一个顶点后，**打印关节向量**（调试阶段）看维数是否等于 **规划自由度**（UR5 手臂常为 6）。

---

## 5. 随机数：为什么要固定种子？

- **调试**：固定 **seed**，每次运行顶点集合 **可复现**，方便对比改边逻辑前后的行为。  
- **提交**：部分评分可能多次运行；**过强的随机性** 可能导致偶发失败 —— 这靠 **足够大的 n** 与 **边上检测** 缓解，不是靠「赌运气」。

建议：用 **`std::mt19937`** + **`std::uniform_real_distribution`**（C++11），种子可用 `std::random_device` 或固定常数（调试时）。

---

## 6. 目标个数 n 怎么取？（经验范围）

| 场景 | 粗调建议 |
|------|-----------|
| 第一次跑通 | **小**（如 100～300），先看 **不崩溃、|V| 满** |
| 简单挡板 | 数百～一千级常够试连通 |
| Bugatti 窄缝 | 往往要 **更大 n** 或配合 **make_edges** 的 k、插值步长（第 5～9 讲） |

**原则**：n 太小 → 图断；n 太大 → **make_edges** 若为 **O(n²)** 或高 k 会 **爆时间**。后面可在 **`solve` 里对失败重试** 时增大 n（与第 8 讲联调一致）。

## 8. 边界与失败情况

| 情况 | 建议行为 |
|------|-----------|
| **max_attempts** 用尽仍 **|V| < n** | 用已有顶点继续，或 **直接判失败**（在 `make_vertices` 或 `solve` 里统一约定） |
| **|V| == 0** | **`solve` 应失败**；勿在空图上做 edges |
| **start 本身碰撞** | 一般 **规划请求无效**；不归咎于 `make_vertices`，但调试时要意识到 |

## 11. 常见误区

- **误区 1**：关节顺序与 **JointModelGroup** 不一致 → 臂的运动与想象不符、碰撞错。  
- **误区 2**：用了 **度** 而库用 **弧度**（或反之）→ 全在奇怪姿态。  
- **误区 3**：**n 极大** 但 **make_edges** 仍是暴力全对 → 编译能过、一 Plan **卡死**。  
- **误区 4**：把 **start / goal** 强行放进 **V** —— 不是不可以，但经典 PRM 常 **分开处理**；若骨架 `solve` 已单独接 start/goal，避免重复逻辑。
